Turn a concept into a new widget in `beshir-widgets` — an independent, iframe-embeddable Preact + Vite SPA deployed to its own `<slug>.widgets.beshir.org` subdomain on Cloudflare Workers. A slow-tier orchestrator copies the repo, runs research → plan → implement → an authoritative in-sandbox build+validate gate → a first-paint render gate → a deterministic journey-capture stage → a multi-lens UX review with a tiered fix loop → review and fix, commits on a branch, and delivers `/out/repo`. The host lands the branch, pushes it, and opens a PR — then hands you the PR link and the render gallery for a quick review.

Repo: `jbeshir/beshir-widgets`, cloned at `/home/jbeshir/code/beshir-widgets`. Each `widgets/<slug>/` is one independent app; `function-plotter` is the reference template. Three repo docs are the source of truth for choices — read them in the pipeline, don't reinvent them:
- `LIBRARIES.md` — the curated, sized library menu and the "How To Choose" guidance.
- `DATA.md` — the per-widget data contract (`static` / `prebake` / `live`).
- `TESTING.md` — the journey/UX-testing contract: the `#widget-ready` and `data-widget-state` markers, the `data-testid` hook expectation, the `journey.json` schema and action vocabulary, the capture matrix, and how the harness runs.

**Default stack:** **Preact core** runtime; **Observable Plot** for chart-like widgets. Optimize for the best result, not bundle size — ~400 KB gzip is a comfortable ceiling, and larger is fine when it's the best fit (show a loading state if it's big or waits on data). For non-chart widgets (simulation, physics, geometry/graphs, animation, drag/interaction), pick the best-in-class tool from `LIBRARIES.md`.

**Watch out:** Every widget must **render offline** in dev and the E2E render (which runs at `egress=none`) — bundle the code, and data widgets ship a committed `sample` (see `DATA.md`). Production may fetch at runtime (live data, map tiles), but the offline render must still pass, so give remote data a loading state. The orchestrator must `cp -a /workspace/repo /out/repo` itself (a `sandbox_script` child writes only to `/out/child/<name>`). `/workspace` is torn down on exit; only `/out` persists.

## The repo's per-widget convention

A widget directory carries: `package.json` + `package-lock.json`, `vite.config.ts` with **`base: './'`** (so the build loads over `file://` for the render and at the subdomain root), `index.html` (with the two favicon `<link>`s — see Favicon below), `src/`, `public/_headers` (removes `X-Frame-Options`/`Content-Security-Policy` so it embeds anywhere), `public/favicon.svg` + `public/favicon.ico` (the browser-tab icon — see Favicon below), `wrangler.jsonc` (`assets` Static-Assets block + a `routes` entry with `custom_domain: true`), `widget.json` (machine-readable metadata: slug/title/description/hostname/dataSources/embed snippet, plus an optional `data` block per `DATA.md`), an optional `journey.json` (the declarative interaction journey the test harness drives — the `state × viewport × scheme` matrix per `TESTING.md`; required for any widget with interaction beyond first paint), and a `README.md`. The slug drives everything: dir basename = `widget.json.slug`, Worker name = `widget-<slug>`, hostname = `<slug>.widgets.beshir.org`. Deployment is declarative — Cloudflare Custom Domains auto-create DNS + TLS on `wrangler deploy`; no DNS scripting.

**Testing hooks (per `TESTING.md`).** Every widget renders an element with `id="widget-ready"` once first paint is done — that is the first-paint gate's assertion. Interactive widgets additionally set `data-widget-state` on `<html>` reflecting their lifecycle (`ready`/`loading`/`populated`/`empty`/`error` as applicable) and expose stable `data-testid` attributes on their controls — the journey harness waits on the state marker (never a settle-timeout) and drives the controls by testid/role. Adding the testids also improves accessibility for free.

**Favicon (the brand tile).** Every widget ships its own browser-tab icon, consistent with the personal-site brand (rose red + gold, after `beshir-personal-website/public/favicon.svg`). The shared element is a **rounded rose-red tile**; the per-widget element is a **gold/cream glyph** evoking what the widget does. Concretely:
- `public/favicon.svg` — the primary, hand-authored, dark-mode-aware. A `viewBox="0 0 32 32"` rounded-rect tile (`rx="7"`) filled with a vertical rose gradient (light `#a4243b`→`#7d1b2b`, dark `#c33049`→`#8f2030`), carrying a centred glyph in warm cream (`#fdf2e0`, dark `#fff4e2`) with a gold accent detail (`#f1c453`, dark `#ffd166`) that echoes the flower's heart. Drive the light/dark swap with a `<style>` `@media (prefers-color-scheme: dark)` block (style the gradient `<stop>`s and glyph fills via classes — see a sibling's favicon). Keep the glyph legible at 16px: one bold idea, generous strokes (~2.5px), no fine detail.
- `public/favicon.ico` — a small multi-size raster fallback (16/32/48) **generated from the SVG's light variant** in the validate step (below), not hand-made. It silences the bare `/favicon.ico` probe and covers legacy browsers.
- `index.html` `<head>` carries both, ico first so modern browsers still prefer the SVG via `sizes="any"`:
  ```html
  <link rel="icon" href="./favicon.ico" sizes="any" />
  <link rel="icon" type="image/svg+xml" href="./favicon.svg" />
  ```
`base: './'` makes the relative hrefs resolve both over `file://` and at the subdomain root. `validate-widgets.mjs` enforces that both `<link>`s and both files are present. The favicon is not part of the render/journey screenshots (it's chrome, not page content), so it has no capture cell — it's a deliverable, not a gated visual.

Three shared scripts already live in the repo and are reused by the gates, not rewritten:
- `scripts/validate-widgets.mjs` — asserts the slug/Worker/hostname/wrangler invariants, validates the optional `widget.json` `data` block, and shape-checks `journey.json` when present.
- `scripts/render.cjs` — CommonJS Playwright harness; loads `file://<dist/index.html>`, waits for `#widget-ready`, screenshots to `/out`. It carries the file:// + ES-module fixes (`--allow-file-access-from-files`, `waitForSelector` state `attached`). This is the cheap first-paint pass/fail gate.
- `scripts/journey.cjs` — the generic, deterministic journey harness. Driven by a widget's `journey.json`, it replays each declared state's steps across the `state × viewport × scheme` matrix, waits on the per-state marker, screenshots each cell, and records console/page errors, horizontal overflow, and an accessibility tree to `journey-report.json`. It freezes `Date`/`Math.random` and disables animation for stable shots, and **exits non-zero** on any error or missed marker — it is both the screenshot generator and an interaction correctness gate.

## Data (per `DATA.md`)

Most visualization widgets are **`static`** (self-contained, no external data). If the concept needs real data, the widget always ships a committed **`sample`** dataset so dev and the offline E2E render work with no egress, and declares a `data` mode in `widget.json`:
- **`prebake`** — the host/CI fetches + normalizes real data into the bundle at deploy time (open-egress lane); the build/validate/render lanes only ever see `sample`. The pipeline produces the `sample` and a `prebake` command; the host runs `prebake` when wiring deploy.
- **`live`** — the deployed Worker fetches at runtime; `sample` is the offline/E2E fallback.

The agent build/validate/render lanes can never reach the open web — only the host/CI or the deployed Worker can. Build and render against `sample`.

## Procedure

1. **Host prep (checkout-free).** Update the remote-tracking ref *without touching the working tree*, so concurrent pipelines never fight over `HEAD` or the index: `git -C /home/jbeshir/code/beshir-widgets fetch origin main`. Do **not** `switch`/`pull` the working tree — the pipeline branches from `origin/main`, not from whatever is checked out, so several widget builds can run and land at once without waiting on each other. Capture the host commit identity to hand to the orchestrator (so the in-sandbox commit is authored correctly and needs no host re-author): read `git -C … config user.name` and `git -C … config user.email`. Derive the new widget's `<slug>` (lowercase, hyphenated) from the concept.

2. **Launch the orchestrator** (slow tier, `agent=claude-code model=opus` when driving Claude) with `directories: ["/home/jbeshir/code/beshir-widgets"]`. It copies the repo whole (incl. `.git`) into `/workspace/repo` and runs steps 3–9 autonomously.

3. **Research** — a `sandbox_research` child (isolated, open egress). It nails the math/algorithm and **finds appropriate data sources** for the concept, or concludes the widget is self-contained. Because research can't see the repo, the orchestrator first reads one or two sibling widgets (e.g. `function-plotter`) and `LIBRARIES.md`/`DATA.md`, and passes their salient shape into the research prompt. → `/out/FINDINGS.md`. For a data widget, the research output must include a representative **`sample` dataset** (so dev/E2E stay offline) and, for `prebake`, the fetch+normalize recipe per `DATA.md`.

4. **Plan** — the orchestrator decides copy-a-template (default: clone `widgets/function-plotter` and rework it) vs scratch, picks the **stack** (Preact core; Observable Plot for charts; otherwise the right tool from `LIBRARIES.md`) and the **data mode** (`static`/`prebake`/`live`), fixes the `<slug>`, and writes `/workspace/PLAN.md` with a handoff contract.

5. **Implement** — one or two medium-tier `sandbox_agent` children (`name=phase01`, …; lowercase DNS-1123 labels). They create `widgets/<slug>/` honouring the convention above — Preact core, the chosen viz library, the `#widget-ready` marker, `base: './'`, `public/_headers`, `widget.json` (incl. the `data` block + committed `sample` for data widgets), `wrangler.jsonc`, and the **favicon** (author `public/favicon.svg` as the brand rose-red tile + a per-widget gold/cream glyph per the Favicon convention above, and add the two `<link>`s to `index.html`; the `.ico` is generated in validate, not here). For any interaction beyond first paint they also add the **testing hooks** (`data-widget-state` lifecycle on `<html>`, `data-testid` on controls) and author **`journey.json`** — a journey covering the canonical states the widget can be in (initial, populated, and as applicable empty/loading/error), driven by the real selectors, per `TESTING.md`. Build it for polish from the start: a clean card, `prefers-color-scheme` dark mode, `ResizeObserver` width, `prefers-reduced-motion`. Phases share `/workspace` — run them sequentially. Each writes `/out/SUMMARY.md`.

6. **Validate** (authoritative gate) — a `sandbox_script` child, `name=validate`, `image=node`, `egress=package-managers`. In `widgets/<slug>/`: first **generate the favicon** `public/favicon.ico` from `public/favicon.svg` (a throwaway `npx -y -p sharp -p png-to-ico node -e …` that rasterizes the SVG's light variant at 16/32/48 with `sharp` and packs them with `png-to-ico`); then `npm install` (generates `package-lock.json` — keep it) then `npm run build` (produces `dist/`, copying `public/favicon.{svg,ico}` into it); then at repo root `node scripts/validate-widgets.mjs` (which asserts the favicon `<link>`s and files are present). All must exit 0. On failure, spawn a fix phase and re-validate.

7. **First-paint gate** — a `sandbox_script` child, `name=render`, `image=browser`, `egress=none`. The build from step 6 left `dist/` in the shared `/workspace/repo`. Run `node /workspace/repo/scripts/render.cjs /workspace/repo/widgets/<slug>/dist/index.html widget-ready /out`. It must exit 0 and write `render-ok` + a non-blank `screenshot.png`. The `browser` image builds on first use (slow first call). If it fails (e.g. `#widget-ready` never appears, or the page is blank), spawn a fix phase, re-validate, re-render. Cap render+fix at 3 rounds.

8. **Journey capture** — a `sandbox_script` child, `name=journey`, `image=browser`, `egress=none`. Run `node /workspace/repo/scripts/journey.cjs /workspace/repo/widgets/<slug>/dist/index.html /workspace/repo/widgets/<slug>/journey.json /out`. It must exit 0, write `journey-ok`, `journey-report.json`, and the per-cell PNG gallery (`<state>__<viewport>__<scheme>.png`). A non-zero exit means a real defect — a missed lifecycle marker, a console/page error, or horizontal overflow in some cell — so spawn a fix phase, re-validate, re-capture (cap 3 rounds). Copy the gallery + `journey-report.json` into `/workspace/repo/.journey-out/` and into `/out/gallery/`. (A purely static widget with no interaction can ship a minimal one-state `journey.json` or skip this stage; anything with interaction must have a journey.)

9. **Multi-lens UX review** — fan out **five parallel** medium-tier `sandbox_agent` reviewers (`name=review-visual|review-responsive|review-theme|review-flow|review-a11y`), each shown the gallery + `journey-report.json`, each a distinct lens, each emitting structured findings `{severity: blocking|should|nice, state, viewport, scheme, issue, suggested_fix, evidence}`: **visual-design** (typography, spacing, colour, hierarchy, polish), **responsive** (same state across viewports — overflow, cramping, touch-target size), **theme** (light vs dark per state — invisible text, contrast, mode-specific breakage), **interaction-flow** (the state sequence — discoverability, feedback, are empty/error states intentional), **accessibility** (the a11y tree + contrast + labels/focus). Reviewers view PNGs visually (Read renders them).

10. **Synthesize + tiered fix loop** — the orchestrator merges the findings, dedups, tiers them, and writes `/out/UX-REVIEW.md`. It then spawns a medium-tier fix `sandbox_agent` that addresses the **blocking** and **should** findings, and re-runs validate → journey on the changed states. Loop fix→validate→journey until no blocking/should findings remain or after 3 rounds; record `nice` items as future polish. This is a first-class gate: a correct-but-ugly or inaccessible widget is not done. Keep the final gallery in `/out/gallery/`.

11. **Review + finalize** — a medium-tier `sandbox_agent` (`name=review01`) reads `git -C /workspace/repo diff`, judges quality, completeness vs the concept, **and visual/UX polish**, and writes `/out/REVIEW.md` ending in `PASS` or `CHANGES_NEEDED` (≤3 rounds; fix and re-validate/re-capture/re-review between rounds). Then: set git identity to the **host identity passed in** (`git config user.name`/`user.email` from host prep), branch `widget/<slug>` from **`origin/main`** (`git switch -c widget/<slug> origin/main` — **never** from the mounted working-tree HEAD, which may sit on an unrelated widget's branch and would contaminate this widget's base), commit the widget (single commit, already authored as the host so the host needs no re-author). The orchestrator itself runs `cp -a /workspace/repo /out/repo`, ensures `/out/gallery/` (final PNGs + `journey-report.json`) and `/out/UX-REVIEW.md` exist, and writes `/out/CHANGES.md` (branch, **base commit = `git rev-parse origin/main`**, slug, hostname, stack + data mode, what the widget does, the gate summary, the journey cell count, how many UX fix rounds ran and what was fixed, the gallery path). Print `DONE`.

## Launching the orchestrator

- **`directories: ["/home/jbeshir/code/beshir-widgets"]` is mandatory.** Without it the orchestrator wakes with no repo.
- Tier: **slow** orchestrator; **medium** implement/review-lens/fix phases; the `validate`/`render`/`journey` children are `sandbox_script` (no tier — `image=node` / `image=browser`).
- Tell it explicitly: **do NOT build, validate, render, or run the journey yourself** — the agent image has no Node toolchain or browser; those are `sandbox_script` children.
- Restate the grounding facts (Workers Static Assets, `custom_domain: true` auto-DNS, iframe `_headers`, `base: './'`, offline render/journey at `egress=none`, the `#widget-ready` + `data-widget-state` markers and `data-testid` hooks, the default stack) and that `LIBRARIES.md`/`DATA.md`/`TESTING.md` are the authority for stack, data, and testing choices.

## Writing the orchestrator prompt

Brief it as a complete document: the concept and **what a good, attractive demo of it looks like**; the **host commit identity** (`user.name`/`user.email`) it must author the final commit with, and that it must branch `widget/<slug>` from `origin/main` (not the mounted HEAD); the per-widget convention, the testing hooks, the favicon convention (brand rose-red tile + per-widget gold glyph, `.ico` generated in validate), the default stack, and the three shared scripts to reuse; the eleven pipeline steps with the child-naming rule and that research is isolated; the explicit VALIDATE (`image=node`, `npm install` + `npm run build` + `node scripts/validate-widgets.mjs`), RENDER (`image=browser` `egress=none`, `scripts/render.cjs … widget-ready /out`), and JOURNEY (`image=browser` `egress=none`, `scripts/journey.cjs … journey.json /out`) commands; the **five-lens UX review fan-out and the tiered fix loop, where the gallery is actually viewed and blocking/should findings are fixed and re-verified**; the data contract (committed `sample`, mode in `widget.json`); the self-contained / offline-render constraint; and the output contract (`/out/FINDINGS.md`, per-phase `/out/SUMMARY.md`, `/out/UX-REVIEW.md`, `/out/REVIEW.md` with a verdict line, `/out/repo` with branch `widget/<slug>`, `/out/CHANGES.md`, `/out/gallery/`, `DONE`).

## Host-side landing + PR

The in-sandbox build+validate+render+journey+UX-review gate is authoritative; the host does not re-run it. The landing is **checkout-free** — it only creates a branch ref and pushes, and never switches the working tree — so it can't collide with another widget's in-flight build or landing, and you can land several widgets in any order without waiting.

**Watch the base:** the sandbox branches from `origin/main` as it was when it started, and a long pipeline can finish after `main` has moved. Before pushing, confirm `git merge-base --is-ancestor origin/main widget/<slug>`; if it isn't, rebase the branch onto current `origin/main` (in a sandbox if it touches widget code, so the gates re-verify against the new base) before opening the PR.

1. Read `/out/CHANGES.md` for the branch (`widget/<slug>`), base commit, and hostname.
2. Create the branch ref locally **without touching the working tree**: `git -C /home/jbeshir/code/beshir-widgets fetch <output_dir>/repo "widget/<slug>:refs/heads/widget/<slug>"` (fallback: if `<output_dir>/repo` is absent, fetch from `<output_dir>/child/<name>/repo`). The commit is already authored as the host identity (set in step 11), so there is no re-author step.
3. Push: `git -C … push -u origin widget/<slug>`.
4. Open a PR with `mcp__github__create_pull_request` (base `main`, head `widget/<slug>`); the body summarises the widget, the stack + data mode, the gate results, and the future URL `https://<slug>.widgets.beshir.org`. `check-widgets.yml` runs on the PR.
5. Give the user the **PR URL** and surface a representative screenshot or two from `/out/gallery/` for the quick review. Merging the PR triggers `deploy-widgets.yml` → live at `<slug>.widgets.beshir.org` (once the repo's one-time Cloudflare secrets are set). For a `prebake` widget, run its `prebake` command (host/CI, open egress) before the first deploy so real data lands in the bundle.

## Output contract

```
/out/
  repo/           # full .git repo; host fetches widget/<slug> from here
  gallery/        # the final journey gallery (<state>__<viewport>__<scheme>.png) + journey-report.json
  FINDINGS.md     # research: algorithm + data sources (or self-contained); sample data for data widgets
  SUMMARY.md      # per implement/fix phase
  UX-REVIEW.md    # synthesized five-lens findings, tiered, with what was fixed
  REVIEW.md       # verdict: PASS or CHANGES_NEEDED
  CHANGES.md      # branch, base commit, slug, hostname, stack + data mode, gate summary, journey cells, UX fix rounds, gallery path
```
